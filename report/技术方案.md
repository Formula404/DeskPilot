# DeskPilot 个人桌面助手技术方案

## 1. 目标与可行性结论

DeskPilot 的目标是做一个带桌面悬浮窗的个人桌面助手：用户可以用自然语言下达任务，系统能够识别当前窗口和上下文，选择 CLI、API、浏览器自动化或桌面 RPA 等方式执行操作，并且可以逐步扩展新应用、新技能和新流程。

整体可行，但不适合一次性做成“万能桌面 Agent”。合理路径是先搭建稳定的 Agent 编排内核、工具注册机制、桌面上下文感知和受控执行框架，再逐步为微信、浏览器、音乐软件、文件系统、Office 等高频场景增加专用工具。

已确认的第一版边界：

- 目标系统：仅支持 Windows。
- 桌面 UI：采用 Tauri。
- 模型接入：使用 OpenAI 接口。
- Agent 模式：采用“意图识别 + LangGraph 状态机 + 受限 OpenAI Tool Calling 循环”的模式。
- 上下文记忆：短期状态走 LangGraph checkpoint，任务日志和长期记忆落 SQLite，第一版长期记忆检索使用 SQLite FTS5。
- OCR：采用本地 OCR 库，第一选择 PaddleOCR，不依赖云 OCR。
- 微信能力：只做用户授权下的 RPA 可见内容采集和滚动采集，不直接读取微信本地数据库。
- 第一版 Demo：优先打通网页操作，再逐步扩展网易云音乐和微信导出表格。
- 插件维护：前期只面向开发者维护，不做普通用户自定义插件市场。

关键判断：

- LangGraph 适合作为核心编排框架，尤其适合把“理解任务、规划步骤、调用工具、确认风险、观察结果、继续执行”拆成可控状态机。
- 桌面悬浮窗采用 Tauri 实现，兼顾轻量、前端生态和 Windows 桌面集成能力。
- 当前窗口识别可行，但不同应用的可见信息差异很大。Windows 上可以通过 Win32 API、UI Automation、OCR、浏览器调试协议等组合实现。
- 对有 CLI/API 的软件优先走 CLI/API；对用户当前浏览器页面优先走 Manifest V3 扩展通道；对独立受控浏览器任务和自动化测试使用 Playwright；对无接口的桌面应用再使用 RPA。这样可靠性和可维护性最高。
- 微信聊天记录抓取是高风险场景，第一版不考虑直接读取数据库，只通过用户打开目标群后的 RPA 可见内容采集、截图 OCR、复制和滚动来实现。
- 第一版先做网页操作，因为浏览器 DOM 自动化和 Playwright 的可验证性更强，适合作为 Agent 工具闭环的基础样板。
- “打开网易云音乐放歌”适合作为第二阶段低风险桌面应用 Demo，可先尝试 URI Scheme/CLI/快捷键/窗口自动化，失败再降级到 RPA。
- RPA 与自动化游戏采集素材的底层逻辑一致，都是“感知屏幕 -> 定位目标 -> 执行动作 -> 观察结果 -> 循环修正”。区别是桌面助手要优先利用 UI Automation、应用 API 和 Playwright，只在必要时使用纯视觉和坐标操作。

## 3. 核心模块设计

### 3.1 Tauri 悬浮窗与设置页

悬浮窗承担即时交互：

- 显示在桌面最前层，支持快捷键唤起和隐藏。
- 输入自然语言任务。
- 展示 Agent 当前计划、正在执行的步骤、需要用户确认的高风险动作。
- 允许一键停止当前任务。
- 打开悬浮窗时立即采集当前前台窗口信息，作为任务上下文传给 Agent。

设置页承担配置管理：

- 模型配置：OpenAI API Key、模型名称、代理地址和调用参数。
- 工具开关：微信、浏览器、网易云音乐、Office、文件系统等。
- 权限策略：哪些工具可自动执行，哪些必须确认。
- RPA 配置：截图频率、OCR 引擎、坐标策略、失败重试次数。
- 用户数据目录：日志、任务历史、导出的表格、缓存文件。

技术选型：

- UI Shell：Tauri。
- 前端：React + Vite + TypeScript。
- 后端：Python + FastAPI。
- 通信方式：Tauri 前端通过 HTTP + SSE 连接本地 Agent 服务；浏览器扩展通过 WebSocket 接收后端采集和动作指令，HTTP 作为兼容/兜底。
- Windows 集成：全局快捷键、置顶悬浮窗、托盘菜单、窗口焦点管理由 Tauri 侧负责。

推荐第一阶段采用 Tauri 做 UI，Python FastAPI 做本地 Agent 服务。LangGraph、Playwright、pywinauto、UI Automation、OCR 等工具留在 Python 侧，整体集成更顺。

### 3.2 本地 Agent 服务

建议单独启动一个本地服务，提供如下接口：

- `POST /chat`：提交用户任务。
- `POST /tasks/{id}/cancel`：取消任务。
- `GET /tasks/{id}`：查询任务状态。
- `GET /events`：通过 SSE 推送执行过程。
- `GET /context/current-window`：获取当前窗口上下文。
- `WebSocket /browser/ws`：浏览器扩展连接通道，用于后端主动请求当前页上下文和下发受控页面动作。
- `GET /tools`：查看已注册工具。
- `POST /settings`：更新配置。
- `POST /approvals/{id}`：提交用户对敏感操作的允许或拒绝。

本地服务负责把 UI 和自动化能力解耦。后续无论悬浮窗换技术栈，Agent 内核都可以保持稳定。

### 3.3 LangGraph 与 Tool Calling Agent

Agent 不建议做成单纯的“意图识别 + 固定 workflow”，也不建议做成完全开放的 ReAct 循环。第一版应采用受限 tool calling 模式：

- 意图识别负责确定任务边界：网页、桌面应用、文件、微信、音乐、问答、系统设置等。
- 每个 intent 定义允许工具白名单、最大步数、风险策略和兜底路径。
- LLM 在白名单工具集合内通过 OpenAI tool calling 自主决定下一步工具。
- 每一步都有 observation、超时、失败处理、权限检查和 task_steps 记录。
- 固定 workflow 只用于必须强约束的高风险场景或兜底路径，不作为普通任务的默认实现。

这样做的原因：

- 纯 workflow 稳定，但大模型不参与决策，项目缺少 Agent 能力。
- 完全开放的 ReAct/tool use 灵活，但桌面操作风险高，容易乱点、误删、重复执行。
- 受限 tool calling 能把“工具边界和安全策略”控制住，把“下一步选择和信息整合”交给模型。

建议把 LangGraph 状态定义为：

- `user_input`：用户原始指令。
- `desktop_context`：当前窗口、应用名、标题、截图、OCR、浏览器信息。
- `intent`：识别后的任务意图。
- `plan`：步骤计划。
- `tool_calls`：已调用和待调用工具。
- `observations`：工具返回结果、截图变化、文件生成结果。
- `risk_level`：风险等级。
- `requires_confirmation`：是否需要用户确认。
- `memory_refs`：本次任务检索到的历史偏好、应用配置、过往经验。
- `approval_state`：当前等待或已完成的用户审批。
- `final_response`：最终反馈。

推荐图结构：

```text
收集上下文
  -> 意图识别
  -> 选择工具白名单和系统提示词
  -> 风险评估
  -> Tool Calling Loop
       -> LLM 选择工具
       -> 权限检查 / 审批
       -> 执行工具
       -> 写入 observation
       -> LLM 判断是否继续
  -> 最终回答 / 失败兜底
```

执行策略优先级：

1. 官方 API 或本地 API。
2. CLI、URI Scheme、快捷命令。
3. 浏览器扩展通道操作用户当前页，例如采集、点击、输入、滚动、抽取表格。
4. Playwright 受控浏览器自动化，用于独立浏览器任务和自动化测试。
5. 系统 UI Automation，例如 Windows UIA。
6. 图像识别、OCR、坐标点击等 RPA。

这个优先级很重要。RPA 最像人操作，但也是最脆弱的方案，应该作为兜底而不是默认路径。

### 3.4 上下文记忆设计

记忆分三层，不把所有内容都塞进模型上下文。

第一层：任务内短期记忆

- 存储在 LangGraph state 中。
- 保存当前用户输入、当前窗口上下文、计划、工具调用、观察结果和审批状态。
- 生命周期是单个任务，任务结束后写入任务日志。

第二层：会话与任务历史

- 存储在 SQLite。
- 保存用户任务、执行步骤、工具结果摘要、失败原因、生成文件路径。
- 敏感内容默认脱敏，例如聊天记录正文默认不进入通用日志，只记录采集数量、文件路径和任务状态。
- 用于“继续上次任务”“查看执行记录”“分析失败原因”。

第三层：长期偏好和经验记忆

- 结构化偏好存 SQLite 表，例如默认导出目录、常用浏览器、常用应用路径、音乐播放器路径、是否自动确认低风险动作。
- 非结构化经验可做向量检索，例如“某个网站登录按钮定位经验”“某个软件搜索框识别方式”。
- 第一版不做向量检索，长期记忆检索固定使用 SQLite FTS5。后续如果需要语义检索，再加入 OpenAI `text-embedding-3-small` 和 SQLite 向量扩展。

推荐存储结构：

```text
SQLite
  settings              用户配置
  task_runs             任务记录
  task_steps            工具调用和观察记录
  approvals             审批记录
  app_profiles          应用路径、窗口规则、RPA 配置
  user_preferences      用户偏好
  memory_items          长期文本记忆
  memory_fts            FTS5 全文检索索引
```

记忆写入策略：

- 默认只自动写入任务日志和低敏配置。
- 长期记忆需要通过规则筛选，例如“用户明确偏好”“重复出现的应用路径”“成功的 RPA 定位策略”。
- 高敏内容不进入长期记忆，除非用户显式允许。

## 4. 工具系统与扩展机制

建议把每个可执行能力做成 Tool Plugin，而不是写死在 Agent 里。第一版不是 MCP，而是本地 Tool Registry + OpenAI tool calling：Tool Registry 是本地执行层，OpenAI tool calling 是模型决策层，中间需要 schema adapter 和名称映射。后续如果需要接入外部工具生态，可以再把部分 Tool Registry 能力封装成 MCP server。

每个工具包含：

- `name`：工具名称。
- `description`：给 LLM 的能力描述。
- `schema`：结构化参数。
- `openai_name`：暴露给 OpenAI 的合法 function name，例如 `browser_collect_current_page`。
- `permissions`：需要的权限。
- `risk`：默认风险等级。
- `handler`：实际执行函数。
- `observe`：执行后如何验证结果。
- `fallbacks`：失败后的降级工具。

示例工具：

```text
music.netease.play_song
  输入: song_name, artist?
  优先级:
    1. URI Scheme 或本地协议
    2. 启动应用并搜索
    3. RPA 点击搜索框、输入歌名、播放

wechat.group.export_visible_messages
  输入: group_name, max_messages, output_path
  优先级:
    1. 用户提供的导出文件解析
    2. 微信窗口可见内容复制
    3. 截图 OCR + 滚动采集
```

插件目录建议：

```text

    tool_calling.py
    prompts.py
  tools/
    registry.py
    base.py
    browser/

    office/
    wechat/

    screenshot.py

    browser_context.py
  safety/

    approvals.py
    audit_log.py
  api/

  ui/
    tauri-app/
```

前期插件只面向开发者维护。插件可以先采用 Python 包和配置文件形式，不需要做面向普通用户的可视化插件市场；等核心工具接口稳定后，再考虑插件安装、禁用、升级和权限声明 UI。

## 5. 桌面上下文识别

打开悬浮窗时需要立即识别当前窗口，建议采集：

- 前台窗口句柄。
- 进程名和可执行文件路径。
- 窗口标题。
- 窗口位置和大小。
- 当前屏幕截图。
- OCR 结果。
- 如果是浏览器，则补充当前 URL、页面标题、DOM 摘要。

Windows 实现路线：

- 前台窗口：`GetForegroundWindow`、`GetWindowText`、`GetWindowThreadProcessId`。
- UI 元素树：Microsoft UI Automation 或 `pywinauto`。
- 截图：`mss`、Pillow。
- OCR：PaddleOCR。
- 浏览器当前页上下文与轻量动作：Chrome/Edge Manifest V3 扩展，通过本地 WebSocket 接收后端指令。
- 受控浏览器自动化：Playwright。

需要注意：当悬浮窗获得焦点后，前台窗口会变成悬浮窗自己。因此要在唤起悬浮窗前或刚唤起时记录“上一个活动窗口”，或者让全局快捷键服务先采集窗口，再打开悬浮窗。

## 6. RPA 能力设计

RPA 层建议分成三种能力：

1. 结构化 UI 自动化：通过 UIA 找按钮、输入框、菜单项。
2. 视觉自动化：通过截图、OCR、模板匹配找目标。
3. 坐标自动化：最后兜底，使用相对窗口坐标，避免绝对屏幕坐标。

本项目的 RPA 和自动化打游戏收集素材类 RPA 是同一套核心逻辑：

```text
截图感知
  -> 图像预处理
  -> OCR / 模板匹配 / 目标检测
  -> 决策下一步动作
  -> 鼠标键盘执行
  -> 再截图验证
```

桌面助手与游戏 RPA 的差别：

- 桌面应用通常可以优先用 UIA、窗口句柄、控件树，稳定性高于纯图像识别。
- 游戏 RPA 往往只能依赖画面、坐标和模板匹配。
- 桌面 Agent 需要权限确认、停止按钮、日志、敏感操作拦截，不能只追求自动执行。
- 桌面应用窗口大小、DPI 缩放和主题会影响视觉识别，所以坐标必须使用窗口相对坐标和锚点定位。

核心动作：

- 启动应用。
- 激活窗口。
- 点击 UI 元素。
- 输入文本。
- 发送快捷键。
- 滚动。
- 等待界面变化。
- 截图和 OCR。
- 验证操作是否成功。

每个 RPA 动作都应该有观察和校验，不建议只做“点击完就认为成功”。

RPA 第一版固定技术栈：

- 截图：`mss`。
- 图像处理：`opencv-python`。
- OCR：`PaddleOCR`。
- 鼠标键盘：`pyautogui`。
- Windows 窗口与进程：`pywin32`。
- UI Automation：`pywinauto`。
- 坐标策略：窗口相对坐标 + 图像锚点，不使用固定屏幕绝对坐标。
- RPA 脚本格式：Python 工具函数，不引入独立商业 RPA 平台。

## 7. 典型场景方案

### 7.1 当前网页执行操作

第一版优先实现网页操作，目标是把“上下文识别 -> Agent 规划 -> 浏览器扩展通道执行当前页采集/动作 -> 观察校验 -> 返回结果”的闭环跑通。Playwright 保留为独立受控浏览器任务和自动化测试能力。

典型能力：

- 识别当前浏览器窗口。
- 获取当前 URL、页面标题、DOM 摘要和可见文本。
- 根据用户指令点击按钮、填写表单、提取表格、下载文件。
- 将网页表格保存为 Excel。
- 总结当前网页内容。
- 对执行结果截图或读取 DOM 进行校验。

推荐技术路径：

1. Chrome/Edge Manifest V3 扩展启动后连接本地 FastAPI 的 WebSocket 浏览器通道。
2. 用户在 DeskPilot 发送任务后，后端通过该通道下发 `collect_page`、`click`、`type`、`scroll`、`extract_table` 等结构化指令。
3. 扩展 background service worker 将指令转发给目标 tab 的 content script。
4. content script 执行当前页采集或轻量 DOM 动作，并通过同一通道回传结果。
5. 对独立浏览器任务、复杂回归测试和扩展无法覆盖的场景，使用 Playwright 启动受控浏览器上下文。
6. 每一步执行后读取 DOM 或截图观察，失败则重试或请求用户确认。

第一版网页 Demo 示例：

- “把当前网页的表格保存成 Excel”。
- “帮我总结当前页面，并保存到本地 Markdown”。
- “在当前网页搜索某个关键词并打开第一个结果”。

### 7.2 获取微信群聊天记录并写入表格

推荐分阶段实现。

第一阶段：用户辅助模式

- 用户打开目标微信群。
- 唤起悬浮窗并输入“保存当前群最近 100 条聊天记录到表格”。
- 系统识别当前窗口为微信。
- 使用复制、截图 OCR 或 UIA 读取当前可见消息。
- 自动滚动并重复采集。
- 去重、按时间排序、结构化为 `时间 / 发送人 / 内容 / 消息类型`。
- 写入 `.xlsx`。

第二阶段：半自动定位群

- 用户输入群名。
- 系统打开微信、搜索群、进入会话。
- 采集记录并保存。

第三阶段：RPA 稳定性增强

- 为微信窗口建立专用 RPA 流程。
- 针对聊天气泡、时间分隔、发送人、滚动边界做模板和 OCR 优化。
- 支持采集前预览和采集后人工复核。
- 明确不绕过加密、不直接读取本地数据库、不无授权读取隐私数据。

风险：

- 微信 UI 变化会影响 RPA。
- OCR 可能识别错误。
- 聊天记录涉及隐私，需要本地处理、权限确认和日志保护。
- 不承诺完整读取历史数据库，只实现用户打开窗口后的 RPA 采集。

### 7.3 打开网易云音乐播放歌曲

推荐执行顺序：

1. 检查是否存在可用 CLI、URI Scheme 或本地协议。
2. 如果应用已打开，激活窗口并搜索歌曲。
3. 如果未打开，启动应用。
4. 使用 UIA 或 RPA 定位搜索框。
5. 输入歌名并播放第一个匹配结果。
6. 截图或窗口状态验证是否进入播放状态。

该场景适合作为早期端到端 Demo，因为风险低、用户体验直观、失败成本小。

该场景建议排在网页操作之后。它可以验证桌面应用启动、窗口激活、快捷键输入和 RPA 兜底能力。

## 8. 安全与权限策略

桌面 Agent 必须有明确安全边界。

建议默认需要确认的操作：

- 读取聊天记录、通讯录、浏览器历史等隐私数据。
- 发送消息、发邮件、发帖、下单、转账、删除文件。
- 修改系统设置。
- 安装软件或执行外部脚本。
- 上传本地文件到外部服务。

建议可以自动执行的操作：

- 打开应用。
- 播放音乐。
- 创建本地文件。
- 读取当前用户明确打开的窗口内容。
- 对本地生成文件做格式转换。

日志建议：

- 保存任务输入、计划、工具调用、结果摘要。
- 敏感内容默认脱敏。
- 用户可清空日志。
- 对微信聊天等隐私任务，默认只记录元信息，不记录完整内容。

聊天记录写表格建议字段：

## 11. 主要技术风险

- 桌面 RPA 稳定性：不同分辨率、缩放比例、主题、应用版本都会影响识别。
- 当前窗口上下文：悬浮窗抢焦点会导致识别对象变成自身，需要提前采集。
- 微信数据合规：第一版只做 RPA 可见内容采集，不绕过加密，不读取本地数据库。
- Agent 幻觉执行：必须用结构化工具和权限策略限制，不允许模型直接执行任意命令。
- 长任务可靠性：需要任务队列、取消、超时、重试和执行日志。
- 多模型成本：视觉、OCR、LLM、网页自动化组合后延迟和成本会增加。

## 12. 推荐第一版技术栈

第一版确定技术栈：

- UI：Tauri + React + Vite。
- 前端语言：TypeScript。
- 前端状态管理：Zustand。
- 前端 UI 组件：shadcn/ui + Tailwind CSS。
- 本地服务：Python + FastAPI。
- Python 版本：Python 3.12。
- Python 包管理：uv。
- Agent 编排：LangGraph + OpenAI Tool Calling。
- 模型接入：OpenAI SDK。
- 模型：主模型使用 OpenAI GPT 系列。
- LangGraph checkpoint：SQLite。
- 任务日志、配置、长期记忆：SQLite。
- 长期记忆检索：SQLite FTS5。
- 浏览器通道：Chrome/Edge Manifest V3 浏览器扩展 + 本地 WebSocket/HTTP。
- 浏览器自动化：扩展用于用户当前页轻量动作，Playwright 用于受控浏览器任务和自动化测试。
- Windows 桌面自动化：pywinauto + pywin32。
- 截图：mss。
- 图像处理：opencv-python。
- 鼠标键盘：pyautogui。
- OCR：PaddleOCR。
- 表格：openpyxl。
- 配置：SQLite + `.env`。
- 日志和任务状态：SQLite。
- 后端日志：Python logging。
- 实时事件推送：SSE。
- 测试：pytest + Playwright trace。

如果当前项目还没有代码，建议先从 Tauri 前端、Python 服务、LangGraph 图、网页操作工具链开始，不要一开始就把所有桌面应用插件做完。

## 13. 当前明确结论

1. 只支持 Windows，不做跨平台抽象优先级。
2. UI 采用 Tauri，悬浮窗、设置页、托盘和快捷键都放在 Tauri 侧。
3. 模型只接 OpenAI 接口，先不做多模型路由。
4. 微信只做 RPA，不直接读取数据库或绕过加密。
5. 隐私数据不要求完全离线处理，但需要在上传或调用外部模型前做明确权限提示。
6. 第一版先实现网页操作，再做网易云音乐，最后做微信导出表格。
7. 插件前期只由开发者维护，先保证工具接口和测试规范稳定。
8. Agent 采用受限 OpenAI tool calling，不采用纯固定 workflow，也不开放无限制自主工具调用。
9. OCR 固定使用本地 PaddleOCR。
10. 上下文记忆固定使用 SQLite，长期记忆检索第一版使用 SQLite FTS5，不做向量检索。
11. 当前网页上下文和用户当前页轻量动作固定通过 Chrome/Edge 浏览器扩展通道完成，Playwright 用于受控浏览器任务和测试。

## 14. 下一步建议

建议下一步直接创建项目骨架：

- `frontend/`：Tauri + React + Vite + TypeScript 悬浮窗和设置页。
- `backend/`：FastAPI + LangGraph + OpenAI SDK。
- `browser-extension/`：Chrome/Edge Manifest V3 扩展，用于连接本地服务、读取和操作当前网页。
- `backend/app/tools/browser/`：网页上下文、扩展通道指令和 Playwright 受控浏览器工具。
- `backend/app/context/`：Windows 当前窗口识别。
- `backend/app/safety/`：权限策略和审批流程。
- `data/`：本地配置、任务日志和导出文件。

第一条端到端链路建议定义为：

```text
用户打开网页
  -> 快捷键唤起 Tauri 悬浮窗
  -> 记录当前浏览器窗口
  -> 用户输入“总结当前网页并保存”
  -> LangGraph 识别为网页任务
  -> 进入 web_page_summary 的受限 tool calling loop
  -> LLM 调用 browser.collect_current_page
  -> 扩展采集并回传 observation
  -> LLM 生成总结并调用 file.write_markdown
  -> UI 显示完成结果和文件路径
```
